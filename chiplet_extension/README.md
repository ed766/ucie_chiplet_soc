# UCIe Chiplet Extension for the Power-Aware RISC-V SoC

This directory refactors the single-die RISC-V project into a dual-die system stitched together with a behavioral UCIe 2.0-style fabric. Die A produces plaintext traffic, packetizes it into CRC-protected FLITs, and consumes the returned ciphertext. Die B aggregates the incoming traffic into AES-128 blocks, encrypts them, and streams results back. A PHY/channel stack models latency, skew, jitter, and probabilistic faults so the two-die topology can be validated before hardening.

## Contents and Layout

- `rtl/` — synthesizable RTL and supporting behavioral models  
  - `soc_chiplet_top.sv` — ties the two dice to the shared channel and exposes monitors  
  - `soc_die_a_top.sv` / `rtl/die_a/die_a_system.sv` — traffic generator, scoreboard FIFO, AES mirror  
  - `soc_die_b_top.sv` / `rtl/die_b/die_b_system.sv` — plaintext aggregator, AES core, return-stream FIFO  
  - `d2d_adapter/` — FLIT packetizer/depicketizer, credit manager, link FSM, retry control, UCIe Tx/Rx, lane interface  
  - `phy_model/` — configurable pipeline + jitter + error injection between adapter and channel  
  - `channel_model/` — reach-dependent skew, crosstalk-induced stalls, probabilistic lane faults
- `sim/` — self-checking SystemVerilog benches and assertion macros  
  - `tb_ucie_prbs.sv` stresses the adapter/PHY/channel chain with PRBS-generated FLITs  
  - `tb_soc_chiplets.sv` runs the full two-die datapath and cross-checks ciphertext on both sides
- `upf/` — UPF 4.0 scaffolding for four power domains (`AON_A`, `PD1_RV32`, `AON_B`, `PD2_AES`) plus the cross-die PST  
- `scripts/` — Python helpers for log parsing (`parse_bw_latency.py`) and report aggregation (`gen_link_reports.py`)  
- `reports/` — CSV artifacts regenerated by the automation (`link_bw_latency.csv`, `link_summary.csv`, stubs for energy/bit)  
- `openlane/` — LibreLane/OpenLane2 configuration stub (`openlane/chiplet/config.json`) targeting `soc_chiplet_top`  
- `Makefile` — entry point for simulations, report refresh, and convenience comparison data

## Architecture

- **Die A (compute chiplet)**:  
  - `die_a_system` synthesizes plaintext words from a running counter, aggregates 128-bit AES blocks, and mirrors the AES core to produce an expected ciphertext FIFO.  
  - `flit_packetizer` groups DATA_WIDTH words into CRC-8 protected FLITs; `flit_depacketizer` validates returning FLITs.  
  - `ucie_tx/ucie_rx`, `credit_mgr`, `link_fsm`, and `retry_ctrl` implement link bring-up, credit accounting, and retry escalation.  
  - `soc_die_a_top` drives the behavioral PHY, presents lane controls, and latches plaintext/ciphertext monitors for debug.
- **Die B (crypto chiplet)**:  
  - `die_b_system` buffers the inbound stream, forms AES blocks, runs `aes128_iterative`, and manages a ring-buffered return FIFO with backpressure to Die A.  
  - Shares the same adapter stack (packetizer, UCIe Tx/Rx, credit manager, retry logic, PHY) to emit ciphertext back across the link.  
  - `soc_die_b_top` exposes a ciphertext monitor for scoreboarding and instrumentation.
- **PHY and channel stack**:  
  - `phy_behavioral` adds user-configurable pipeline depth, jitter insertion, and stochastic bit flips before the traffic enters the channel.  
  - `channel_model` applies reach-based skew, crosstalk stalls, and probabilistic lane faults to emulate an interposer or organic substrate path.  
  - Link faults propagate back through the adapters to exercise CRC/retry machinery.
- **System top (`soc_chiplet_top`)**:  
  - Instantiates both dice, cross-connects them through the channel, and fans out monitors (`plaintext_monitor`, `ciphertext_monitor`, `die_b_ciphertext_monitor`, `crypto_error_flag`) used by the benches and higher-level diagnostics.

## Power Intent (UPF)

This extension keeps power intent in UPF to mirror the base SoC style and to document cross-die low-power behavior without pushing power policy into RTL.

- `upf/die_a.upf` and `upf/die_b.upf` define the **always-on vs switchable** domain split (`AON_A`/`PD1_RV32`, `AON_B`/`PD2_AES`) and provide placeholder supplies so each die can be elaborated independently.  
- `upf/pst_chiplet.upf` defines a **cross-die power-state table** (`RUN`, `SLEEP`, `CRYPTO_ONLY`, `DEEP_SLEEP`) that encodes allowed domain combinations across the two dice.  
- The files are **structural scaffolding**: tool-specific supply sets, power switches, isolation nets, and retention strategies still need to be inserted before power-aware sign-off.

### Techniques and Why It Matters

- **Domain partitioning** makes always-on logic (sideband, control) explicit and lets the compute/AES blocks be power-gated cleanly.  
- **Isolation/retention hooks** (demonstrated in `base_soc/upf/`) are the intended mechanism to clamp outputs and preserve AES key state across PD2 power-down events.  
- **PST-driven intent** provides a single source of truth for allowable die states, which is especially important for UCIe bring-up where link training depends on both ends being powered.  
- **Separation of intent from RTL** keeps functional logic stable while enabling later power-aware simulation, synthesis, and sign-off tooling.

## Verification and Metrics

- `sim/sva_macros.svh` supplies lightweight assertion/timeout macros shared by both benches.  
- `tb_ucie_prbs.sv` checks FLIT integrity through the adapter/PHY/channel stack and reports PRBS error counts.  
- `tb_soc_chiplets.sv` runs the full two-die loop, watching plaintext/ciphertext monitors and asserting on mismatches or unexpected error flags.  
- Logs land in `logs/` (or stubs with the default flow) and feed the Python parsers to generate link bandwidth, latency, and summary CSVs under `reports/`.  
- The benches expect a simulator with solid SystemVerilog support (`SIM_TOOL=iverilog` is the first supported option; Verilator requires additional interface refactoring).

## Automation

From `chiplet_extension/`:

```bash
# Emit stub logs (default SIM_TOOL=stub) and refresh reports
make chiplet-sim
make chiplet-report

# Run real benches once Icarus Verilog is installed
make chiplet-sim SIM_TOOL=iverilog

# Run seeded regressions (requires iverilog)
make regress N=50

# Inject a known bug to confirm the checkers flag failures
make chiplet-sim SIM_TOOL=iverilog SIM_DEFINES='-DUCIE_BUG_CREDIT_OFF_BY_ONE'

# Generate placeholder comparison CSV after reports refresh
make compare

# Clean build/log artifacts
make clean
```

- `chiplet-report` invokes `parse_bw_latency.py` and `gen_link_reports.py` to rebuild link KPI CSVs.  
- `regress` runs seeded PRBS stress tests plus directed corner cases and logs a summary to `reports/regress_summary.csv`.  
- `compare` seeds `reports/compare.csv` with example metrics to illustrate how the chiplet numbers stack up against the single-die baseline.  
- Each target is simulator-agnostic: if `SIM_TOOL` is not supplied, the flow emits deterministic stub logs so the reporting scripts still run.

## Physical-Design Hook

- `openlane/chiplet/config.json` points LibreLane/OpenLane2 at `soc_chiplet_top` and enumerates the RTL required for synthesis.  
- The config uses `soc_chiplet_top.sdc` and a relaxed `CLOCK_PERIOD` (200 ns) for exploratory runs; tighten the clock and constraints for realistic timing closure.  
- Invoke LibreLane from its repository, supplying your PDK root, e.g.:

  ```bash
  python3 -m librelane \
    --pdk-root ~/.ciel/ciel/sky130/versions/<sky130_rev> \
    ~/ucie_chiplet_soc/chiplet_extension/openlane/chiplet/config.json
  ```

- SystemVerilog interfaces (`ucie_lane_if.sv`) and behavioral constructs may need flattening or replacement with synthesis-friendly wrappers before sign-off flows will accept the design. The current top-level wiring uses direct signals; reintroduce interfaces only if you plan to flatten them for OpenLane/OpenROAD.

## Limitations and Notes

- Behavioral models focus on functionality and diagnostics; no timing-calibrated PHY or channel parameters are provided yet.  
- Credit thresholds, FIFO depths, retry holdoff constants, and error injection knobs are tuned for simulation convenience rather than silicon accuracy.  
- UPF files lack detailed supply/port declarations, isolation cells, and retention definitions; integrate tool-specific directives before tape-out.  
- Testbenches assume perfect simulator support for SystemVerilog interfaces and packages; alternative simulators may require adjustments to the adapter/PHY bindings.
- LibreLane runs complete end-to-end with a relaxed clock target, but timing closure at tighter clocks is not representative yet and slow-corner max slew/max cap warnings remain.
